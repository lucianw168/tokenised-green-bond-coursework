# Tokenised Green Bond Coursework Prototype

This repository contains a simplified proof-of-concept for a compliant tokenised green bond, not a production-ready regulated issuance platform.

The prototype implements core lifecycle controls referenced in the report: whitelist-based holding, controlled issuance, restricted transfer, pause, and redemption/burn.

## Project Overview

This project is a coursework submission for UCL exploring how tokenisation can be applied to green bonds in a Hong Kong regulatory context. It provides a single Solidity smart contract (`GreenBondToken`) that models a permissioned fungible token representing a HKD-denominated tokenised green bond, deployed on an EVM-compatible testnet.

## Why a Fungible Token

Green bonds are debt instruments with uniform terms — each unit carries identical rights to coupon and principal. An ERC-20 fungible token is therefore the natural digital representation. Unlike NFTs (ERC-721), a fungible token allows fractional holdings and straightforward secondary transfer among approved holders, which aligns with how institutional bond markets operate.

## Prototype Scope

This prototype demonstrates a **minimal compliant lifecycle** for a tokenised green bond. It is intentionally limited to core controls that can be meaningfully implemented, tested, and verified within a coursework scope.

## Core Features Implemented

| Feature | Description |
|---|---|
| **Whitelist** | Only addresses approved by the issuer/admin can receive and hold tokens. |
| **Mint (Issuance)** | Only the owner/admin can mint tokens to whitelisted investors (primary allocation). |
| **Restricted Transfer** | Transfers to non-whitelisted addresses revert automatically. |
| **Pause / Unpause** | The owner can halt all transfers for compliance or emergency reasons. |
| **Redeem / Burn** | Token holders can redeem (burn) their tokens to simulate bond maturity redemption. |
| **Events** | `InvestorWhitelisted`, `BondIssued`, `Redeemed`, `Paused`, `Unpaused` emitted for on-chain auditability. |

## What Is Not Implemented

| Feature | Reason |
|---|---|
| Coupon distribution engine | Conceptual servicing layer — described in the report but not implemented in code. Implementing an on-chain coupon schedule with real payment logic exceeds coursework scope. |
| Front-end UI | Not required; all interaction is via scripts and tests. |
| Multi-contract architecture | A single contract is sufficient to demonstrate core lifecycle controls. |
| Force transfer / recovery | Optional enhancement; not core to the minimum viable compliance model. |
| Real KYC/AML integration | Off-chain process in practice; the whitelist simulates the on-chain result of KYC clearance. |

## How to Install and Run

### Prerequisites

- Node.js ≥ 18
- npm or yarn

### Install

```bash
git clone <repository-url>
cd tokenised-green-bond-coursework
npm install
```

### Compile

```bash
npx hardhat compile
```

### Run Tests

```bash
npx hardhat test
```

## How to Deploy

1. Copy `.env.example` to `.env` and fill in your RPC URL.
2. Create **4 wallets** in MetaMask (Deployer, Investor A, Investor B, Unapproved). Export each private key into `.env`.
3. Fund **all 4 wallets** with testnet ETH from a faucet (each needs a small amount for gas).
4. Run:

```bash
npx hardhat run scripts/deploy.js --network sepolia
```

5. The script outputs the contract address and writes `deployments/sepolia.json`.
6. Auto-fill all documentation placeholders:

```bash
npm run fill-docs:sepolia
```

7. Verify source on Etherscan:

```bash
npx hardhat verify --network sepolia <CONTRACT_ADDRESS> "HK Green Bond Token" "HKGBT" <MATURITY_TIMESTAMP> <OWNER_ADDRESS>
```

## Demo Scenario

After deployment, run the full lifecycle demonstration:

```bash
npx hardhat run scripts/demo.js --network sepolia
```

The demo executes 8 steps in order (see [docs/demo-scenario.md](docs/demo-scenario.md)):

1. *(Deploy — already done)*
2. Whitelist Investor A
3. Mint (issue) tokens to Investor A
4. Transfer to unapproved wallet → **reverts**
5. Whitelist Investor B
6. Transfer A → B → **succeeds**
7. Pause → transfer → **reverts**
8. Unpause → Redeem/burn → balance and supply decrease

## Limitations

- This is a **coursework prototype**, not a production-ready regulated issuance platform.
- The whitelist mechanism simulates the on-chain outcome of off-chain KYC/AML — it does not perform actual identity verification.
- Coupon distribution is discussed conceptually in the report but not implemented in code.
- The maturity timestamp is illustrative; no automatic redemption trigger is enforced.
- Only tested on EVM testnets; no mainnet deployment is intended.
